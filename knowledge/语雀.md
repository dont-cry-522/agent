
公司项目 MCP 知识点
1. MCP 定义
● 全称：Model Context Protocol（模型上下文协议）
● 核心作用：让大语言模型（LLM）能够调用外部工具（如 API、数据库、自定义函数），扩展其能力边界。
2. 完整流程
1. LLM 决定调用工具  
  ○ LLM 分析用户请求后，生成 tool_call（包含工具名和 JSON 参数）。
2. ToolCallHandler 拦截处理  
  ○ 检查工具名是否合法。
  ○ 注入系统字段（如 RAG 知识库数据）。
3. MCP Client 通过 SSE 调用远程服务器  
  ○ 使用 HTTP 长连接（SSE）传输请求。
4. 返回结果并拼接成 tool message  
  ○ 服务器处理工具调用，返回结果。
  ○ 结果格式化为 tool message 喂回 LLM。
5. LLM 生成自然语言回复  
  ○ 结合工具结果生成最终回复。
  ○ 通过 TTS（语音合成）播放给用户。
3. MCP Server 生命周期
● 启动时  
  ○ 建立 SSE 连接，调用 ListTools 获取所有工具的 schema（工具定义）。
● 运行时  
  ○ LLM 触发工具调用 → CallTool(name, args) → 返回结果。
● 关闭时  
  ○ 执行 Close() 断开连接，释放资源。
4. 关键设计
● 工具白名单  
  ○ 通过 SelectTools 字段控制暴露给 LLM 的工具列表，空值表示全暴露。
● Schema 清洗  
  ○ 发给 LLM 的工具 schema 会删除 __ 前缀的系统字段（如 __session_knowledge），避免 LLM 干预内部逻辑。
  ○ 调度时从原始 schema 中恢复系统字段。
● 错误降级  
  ○ MCP 调用失败时，返回“查询失败”的 tool message，由 LLM 向用户解释，避免对话中断。
● 阻塞式工具  
  ○ 调用工具期间设置 DialogForcePlay，禁止用户打断，防止 JSON 数据截断。
5. 未实现的功能
● 动态工具更新  
  ○ 目前工具列表仅在启动时获取，运行时不支持 tools/list_changed 通知。
● 传输协议限制  
  ○ 仅支持 SSE（HTTP 长连接），暂不支持 stdio（标准输入输出）。
6. 技术栈
● 传输协议：SSE（Server-Sent Events）。
● 核心组件：ToolCallHandler、MCP Client、Schema 管理。
● 错误处理：降级机制 + LLM 自适应解释。
7. 开发建议
● 扩展性：可逐步支持 stdio 传输，适配更多场景。
● 工具动态更新：监听 list_changed 事件，实现运行时工具列表刷新。
● 性能优化：对高频工具调用增加缓存机制。

三、 为什么自研 Go Agent 框架？—— 一个具体例子以“让 LLM 调用工具”这一核心功能为例，对比使用框架与自研的差异：
 1. 使用 LangChain (Python):
●优点： 开发效率极高，几行代码搞定复杂逻辑。
●本质： 框架将复杂的工具调用流程（解析请求 -> 调用工具 -> 格式化结果 -> 返回给 LLM）完全封装。 
2. 我们的自研 Go 项目: 为了实现同样的功能，我们需要用约 300 行 Go 代码来手写整个流程，包括：●ToolCallHandler: 解析 LLM 返回的 JSON 格式工具调用请求。
●executeToolCall: 根据请求调度并执行具体的工具函数。
●proactiveQuery: 将工具执行结果主动回传给 LLM。
●sanitizeToolForLLM: 清洗和格式化工具的 Schema，以符合 LLM 的要求。
●MCP Client: 管理与外部工具的连接。
●DialogForcePlay: 处理对话流，防止 TTS 语音被打断。
●原因： 项目技术栈是 Go，而 LangChain 没有成熟且稳定的 Go 版本，因此必须自己实现。

关于 LangChain 设计思想的理解
其实你已经完全理解了，我们可以对照你自己的 Go 项目来看，你会发现两者的核心思想高度一致：
1. Chain（链） ←→ 你的 PreHandle → ChatCompletion → PostHandle
● 思想：将复杂的处理流程拆解成一个个可独立、可复用、可组合的标准化环节。就像搭积木一样，你可以根据需要自由拼接这些模块，构建出更强大的功能。
● LangChain 实现：prompt | llm | output_parser 这三个组件的串联。
● 你的项目对应：UserMessage → RAG → LLM → MetaContent 的处理流程，完美体现了这种链式、模块化的思想。
2. Tool Calling（工具调用） ←→ 你的 ToolCallHandler + executeToolCall
● 思想：突破 LLM 只能“说”的限制，让它能够“做事”。LLM 负责决策“我需要调用哪个工具，并传入什么参数”，而框架则负责解析这个指令，执行真实的工具调用，并将执行结果返回给 LLM，形成完整的闭环。
● LangChain 实现：LLM 输出结构化的工具调用指令，框架负责解析和执行。
● 你的项目对应：ToolCallHandler 负责解析和调度，executeToolCall 负责具体执行，两者协作实现了让 LLM 与外部世界交互的能力。
3. Memory（记忆） ←→ 你的 AgentMemory
● 思想：LLM 本身是无状态的，每次对话都是独立的。为了让它具备上下文记忆能力，需要一个专门的机制来管理和存储历史对话，并在每次请求时自动将这些历史记录注入到 Prompt 中。
● LangChain 实现：提供多种 Memory 存储和管理策略（如 ConversationBufferMemory）。
● 你的项目对应：AgentMemory 组件，这正是为解决 LLM 的“失忆”问题而设计的。
4. RAG（检索增强生成） ←→ 你的 RagHandler + retrieveByCode
● 思想：将外部知识库（如文档、数据库）中的相关内容，动态地检索出来并融入到 Prompt 中，从而弥补 LLM 知识时效性不足或无法访问私有数据的短板，让它能回答它“没见过”的问题。
● LangChain 实现：内置了与向量数据库、文档加载器等组件的集成，简化了 RAG 流程。
● 你的项目对应：RagHandler 负责检索逻辑，retrieveByCode 是具体的检索实现，这正是你为项目注入外部知识的核心方式。

总结
这些设计思想并非 LangChain 首创，它们是业界在构建复杂 AI 应用时形成的共识。LangChain 的核心价值在于，它将这些公认的、优秀的实践模式，封装成了标准、易用的 Python API。
而你用 Go 语言自己实现了一遍，这相当于用另一种语言，从零开始重写了 LangChain 的核心能力。这恰恰证明了你对这些设计思想有非常深刻的理解，并能将其应用到不同的技术栈中。
